# X-Change AI業務管理システム

**吉田自動車工業 X-Changeタイヤ販売店舗向け プロトタイプ**

> 補助金申請前 説明資料 兼 プロトタイプ仕様書 (v3.15 / 12項目テンプレート版)
> **店舗側はスマートフォン操作 / 本社側はPC操作を想定。スマホでも本社側の確認が可能。**
> **売上音声入力はスマホ標準キーボードのマイクを利用するため、音声認識API料金は不要です。**

## 2. 現在の業務課題

吉田自動車工業様の X-Change タイヤ販売店舗の運営において、従来の業務には以下のような課題があります。

- **拠点分散**: タイヤ販売店舗は本社とは別拠点で運営されている
- **少人数運営**: 店長1名で店舗運営を行っている
- **手作業中心**: 売上報告が紙伝票・Excel での手作業になりやすい
- **レシート管理の遅延**: 経費レシートの提出や確認が遅れやすい
- **本社確認の遅延**: 本社側で売上・経費の確認に時間がかかる
- **月次決算の遅れ**: 月次決算や利益把握が翌月以降にずれ込む
- **利益の見えにくさ**: タイヤ販売部門単体の利益がリアルタイムに見えにくい

これらの課題を、AI と OCR を活用した業務管理システムで解決します。

## 3. 導入後の業務フロー

### 3-1. 売上フロー

```
[店舗] 1. 店長が売上内容を 音声 または テキスト で入力
        ↓
       2. AI が売上項目（商品名 / サイズ / 数量 / 単価 / 合計 / 支払方法 / 車種等）に自動分解
        ↓
       3. 店長が画面で内容確認・必要に応じて修正
        ↓
       4. 「この内容で登録」ボタンで売上登録
        ↓
[本社] 5. 売上一覧に 即時反映（未確認）
        ↓
       6. 本社事務が内容確認 → 「確認済み」 へ
        ↓
       7. 内容に問題があれば 「修正依頼」 をコメント付きで送信
        ↓
[店舗] 8. 修正依頼を受信 → 内容を修正して 「再提出」 → 再び未確認へ
```

### 3-2. 経費フロー

```
[店舗] 1. 店長が レシート画像 をアップロード（撮影 or サンプル選択）
        ↓
       2. OCR がレシートから 日付 / 購入先 / 金額 / 消費税 / 支払方法 / 内容 を読取
        ↓
       3. AI が 12 カテゴリから 経費科目を候補分類（Top3 + 全候補）
        ↓
       4. 店長が内容確認・必要に応じて修正
        ↓
       5. 「この内容で登録」ボタンで経費登録
        ↓
[本社] 6. 経費一覧に 即時反映（未確認、AI分類科目のみ確定済み）
        ↓
       7. 本社事務が 「本社確定科目」 を最終決定（必要なら変更）
        ↓
       8. 「確認済み」 へ
        ↓
       9. 内容に問題があれば 「修正依頼」 をコメント付きで送信
```

### 3-3. 月次フロー

```
[本社] 1. 月次集計画面で売上・経費を確認
        ↓
       2. 未確認・修正依頼中のデータを処理（確認済み / 差し戻し）
        ↓
       3. 月次集計（売上合計・経費合計・粗利益・件数・内訳）を確認
        ↓
       4. 売上CSV / 経費CSV / 月次集計CSV を出力
            ↓ （会計ソフト連携・補助金申請資料に利用）
       5. 「月次締め」 ボタンでその月のデータを 「月次処理済み」 へロック
```

## 11. クイックスタート（プロトタイプ動作確認）

```bash
# このフォルダを開いて index.html をブラウザで開くだけ
open index.html              # macOS

# もしくは簡易ローカルサーバ
python3 -m http.server 8080
# → http://localhost:8080/
```

**動作確認推奨環境:**
- 店舗側: スマートフォン or Chrome DevTools のモバイルエミュレータ（375×812 等）
- 本社側: PC ブラウザ幅 1280px 以上（左サイドバー + テーブル一覧）
- 補助金申請資料用スクリーンショット: PC 全幅 1280px 以上での撮影を推奨

**プロトタイプの特徴:**
- データは `localStorage` に保存され、ブラウザを閉じても保持
- フッター 「**デモデータ初期化**」 でいつでも初期状態に戻せる
- 画面右上の 「店舗 / 本社」 で同じデータを別UIから確認可能（即時反映）
- 依存ライブラリなし。素のHTML/CSS/JSのみで動作

### 11-1. スマートフォン表示対応（v3.9 強化）

本プロトタイプは **スマホ・タブレット・PC のすべてに対応** しています。

| 利用シーン | 想定デバイス | 表示の特徴 |
|---|---|---|
| **店舗側** | **スマートフォン操作を想定** （iPhone / Android, 360〜414px 縦向き） | KPIカード・大ボタン・モーダルすべてがスマホ片手操作に最適化 |
| **本社側** | **PC操作を想定** （1280px〜）/ ただし **スマホでも確認可能** | PCではテーブル一覧、スマホでは自動でカード表示に切り替え |

**スマホ表示時の主な対応:**
- **横はみ出し完全防止** (`body { overflow-x: hidden }`)
- 上部「店舗／本社」切替ボタンが 360px 級画面でも崩れない（ブランド副題は自動で省略）
- 売上一覧／経費一覧のテーブル → **モバイルで自動的にラベル付きカード表示** に変換
- 全モーダル → スマホで `max-width: 95vw` に収まる
- 月次集計画面のKPIカード／内訳カード／月次締めボタンを縦並び化
- 店舗ダッシュボードの 5 KPI を 2列グリッド + 5枚目全幅レイアウトに変更
- 録音マイクボタン・大型サマリー・アクションボタンの縮小調整
- 長文セルは省略表示（`title` 属性でホバー全文確認可能）
- iPhone 390px / Android 360-412px の縦向きで右端切れなし

### 11-2. 売上音声入力 — スマホ標準キーボードのマイクを利用（v3.11 / v3.13 / v3.14 強化）

> 📌 **音声入力の話し方ガイド (v3.14)**
>
> 売上音声入力は、**番号を読み上げる必要はありません**。項目名に続けて短く区切って話すことで、AI分解の精度が上がります。
> **車両番号は音声認識で誤認識しやすいため、登録確認画面で必ず確認・修正してください。**

#### おすすめの話し方（番号なし・項目名形式）

各項目を **「項目名 → 区切り → 値」** の形式で短く区切って話します:

```
売上区分:新品タイヤ販売
商品名:ヨコハマ アイスガード
タイヤサイズ:195/65R15
数量:4本
車種:プリウス
車両番号:品川300あ1234
合計金額:68,000円
支払方法:現金
作業内容:タイヤ交換、バランス調整、廃タイヤ処分あり
備考:高橋様
```

3種類の入力ボタンを用意しています:

| ボタン | 動作 |
|---|---|
| 📱 **スマホ音声入力を使う** | テキスト欄にフォーカス → スマホキーボード起動 → マイク🎤で音声入力 |
| 📋 **サンプル音声を入れる** | デモ用の完全な音声サンプル文を投入（全項目入り） |
| 📝 **入力テンプレートを入れる** | 項目名のみのスケルトンを投入 → 各項目に音声入力で値を埋める |
| クリア | テキスト欄を空にする |

#### AI分解が対応する表現パターン（v3.14）

`parseStructuredFields()` で以下のラベル + 区切り を認識して各フィールドに自動振り分けます:

| 認識する項目名 | 振り分け先 |
|---|---|
| 売上区分: / 売上区分、 | salesCategories |
| 商品名: / 商品名、 | productName |
| タイヤサイズ: / サイズ: / 〜、 | tireSize |
| 数量: / 数量、 | qty |
| 車種: / 車種、 | carModel |
| 車両番号: / ナンバー: / 〜、 | carNumber |
| 合計金額: / 金額: / 〜、 | total |
| 支払方法: / 支払い: / 支払: / 〜、 | paymentMethod |
| 作業内容: / 作業内容、 | workContent |
| 備考: / 備考、 | note |
| お客様: / お客様名: / 〜、 | customer |

区切り文字は **「:」「：」「、」「,」** のいずれにも対応。値は次のラベル/「。」/改行/末尾まで。

**自由文章形式（ラベルなし）も従来通り分解可能:**
> 「新品タイヤ4本、195/65R15、プリウス、合計48,000円、現金、廃タイヤ処分あり」 → 既存の正規表現パーサで処理

#### 入力テンプレート（v3.15: 12項目・日付/店舗/担当者は除外）

**売上音声入力画面を開いた時点 / 「📝 入力テンプレートを入れる」ボタン / 「クリア」ボタン** いずれの場合も、以下の **12項目テンプレート** が初期表示されます:

数量:
単価:
合計金額:
支払方法:
車種:
車両番号:

各行をタップしてカーソルを置き、スマホキーボードのマイクボタンで該当項目だけ短く話す → 音声認識精度が上がる、という流れです。

#### 確認画面（売上 登録前確認）の仕様

- **画面上部の注意バナー（赤系）**: 「🤖 音声入力の内容をAIが分解しました。登録前に、金額・数量・支払方法・車両番号を必ず確認してください。」
- **大型サマリー（青グラデ）**: 合計金額（38px）/ 数量 / 単価 すべてに ⚠️タグ付き
- **支払方法**: 6タイル選択（⚠️ 要確認 タグ）
- **タイヤサイズ・車両番号**: 個別フィールドラベルに ⚠️ 要確認 タグ
- **車両番号**: 注意書き「車両番号は音声入力で間違いやすいため、必要に応じて手入力で修正してください。空欄のままでも登録できます。」

- 「🎤 音声入力へ戻る」を押すと音声入力画面に戻り、元の音声テキストを再編集できる

#### 「✓ 確認して登録」ボタンの動作

- 押すとデータが店舗側 [本日売上一覧] / 本社側 [売上一覧] / [未確認一覧] に **即時反映**
- 確認ステータスは「未確認」で登録される
- バリデーション: 合計金額・数量・支払方法・売上区分（1つ以上）が必須
- 車両番号は空欄でも登録可



> 💡 **本システムの売上音声入力は、スマートフォン標準キーボードの音声入力を利用します。**
> **外部音声認識APIは使用しないため、音声認識API料金は発生しません。**
>
> ⚠️ **iPhoneの場合**: 「設定」 > 「一般」 > 「キーボード」 > 「**音声入力を有効にする**」 が **ON** になっている必要があります。
>
> ⚠️ **Androidの場合**: Gboard 等の標準キーボードに音声入力ボタン（🎤）が表示されていることをご確認ください。

売上音声入力画面（[S2]）は、**スマホのキーボードに標準搭載されているマイクボタン** を使った音声入力をメイン導線として設計しています。

#### 利用フロー（実機スマホで操作する場合）

1. 店舗ダッシュボード → 「🎤 音声で売上登録」
2. 画面上部の案内：「📱 下の入力欄をタップするとスマホのキーボードが開きます。キーボード内のマイクボタン 🎤 を押して売上内容を話してください。」
3. 入力例を確認:「新品タイヤ4本、195/65R15、プリウス、合計48,000円、現金、廃タイヤ処分あり、バランス調整あり」
4. **「📱 スマホ音声入力を使う」ボタン** をタップ、または **大きなテキスト欄を直接タップ**
5. スマホのキーボードが立ち上がるので、**キーボード上のマイクアイコン 🎤** を押す
6. 売上内容を話す → スマホ標準音声認識でテキストに変換 → 入力欄に自動的に文字が入る
7. 入力されたテキストを画面で確認・必要なら手入力で修正
8. 下部の「🤖 AIで分解する」ボタンを押す → 確認画面で項目別表示・編集 → 登録

#### 設計の利点（API料金ゼロの仕組み）

| 項目 | 内容 |
|---|---|
| **音声認識API料金** | **¥0**（端末標準の Google 音声入力 / iOS Siri 音声入力を使用） |
| **APIキー管理** | **不要**（外部API接続なし） |
| **追加SDK・ライブラリ** | **不要**（HTML 標準の `<textarea>` のみ） |
| **個人情報のサーバー送信** | **発生なし**（音声データは端末→OSの音声認識→テキスト返却で完結） |
| **オフライン対応** | スマホOSの音声認識が対応している範囲で利用可（OS依存） |

#### テキストエリアの属性

| 属性 | 値 | 効果 |
|---|---|---|
| `lang` | `"ja-JP"` | 日本語入力を明示、音声認識の精度向上 |
| `inputmode` | `"text"` | 標準キーボードを呼び出す（マイクボタン付き） |
| `enterkeyhint` | `"done"` | モバイルキーボードの確定キーを「完了」表示 |
| `autocomplete` | `"off"` | 自動補完を無効化（音声入力に集中） |
| `font-size` | `16px` | iOS Safari の自動ズームを防止 |
| `rows` | `8` | 大型表示で誤タップ防止 |

#### 主要ボタン (v3.11)

| ボタン | 役割 |
|---|---|

## 14. 技術詳細・改修履歴

### 13-1. データ保存

すべて `localStorage` のキー `xchange.records.v1` に **配列** で格納。レコードは `type: "sale" | "expense"` で識別。

**確認ステータス**（4種）:
- 未確認 / 確認済み / 修正依頼 / 月次処理済み
- 旧名 (承認待ち / 承認済み) は `loadAll()` で起動時に自動マイグレーション

### 13-2. モック範囲

| 機能 | 動作 |
|---|---|
| 音声録音 | マイクボタン押下→1.2 秒後にサンプル文を自動投入 |
| AI項目分解 | 正規表現ベースの日本語パーサ ([app.js parseVoiceText()](app.js)) |
| OCR | 7種のサンプルレシート（ENEOS/カインズ/タイヤ仕入/廃タイヤ/電気/通信/広告）から固定OCRテキスト |
| AI経費分類 | キーワードベースのスコアリング ([app.js classifyExpense()](app.js)) |
| 即時共有 | 同一 localStorage を 2 ロールで共有 → ロール切替で同期表示 |

### 13-3. CSV出力フォーマット

| ファイル名 | 列数 | 内容 |
|---|---:|---|
| `xchange_sales_<YYYY-MM>.csv` | 15 | 日付/店舗/担当/売上区分/商品名/サイズ/数量/単価/合計/支払/車種/車両番号/作業内容/備考/ステータス |
| `xchange_expenses_<YYYY-MM>.csv` | 12 | 日付/店舗/担当/購入先/内容/金額/消費税/支払/AI分類/本社確定/備考/ステータス |
| `xchange_monthly_summary_<YYYY-MM>.csv` | 6ブロック | サマリー(10項目)+粗利計算+売上内訳4種+経費内訳3種+売上全明細+経費全明細 |

すべて UTF-8 BOM 付きで Excel 互換。

### 13-4. 月次締めステータス（5状態）

| ステータス | 条件 | 月次締め可否 |
|---|---|---|
| データなし | レコード0件 | × |
| 確認待ち | 未確認/修正依頼が残っている | × ブロック |
| 月次締め可能 | 全て確認済み | ○ |
| 月次締め進行中 | 確認済み + 月次処理済み混在 | ○ |
| 月次締め完了 | 全て月次処理済み | × （解除のみ可） |

## 15. 補足: 会計処理 / 月次決算 / 補助金申請への活用

### 会計ソフト連携
- 売上CSV / 経費CSV をそのまま freee / マネーフォワード / 弥生 へ取り込み可能
- 本社確定科目が会計仕訳の科目と直結

### 月次決算
- 月次集計CSV 1 ファイルで PL（粗利益）と 内訳（科目別・支払方法別・商品別）を確認
- 月次締め処理でその月のデータを「月次処理済み」にロックし、改竄を防止

### 補助金申請資料
- 月次集計画面のスクリーンショット（5セクション）が、売上 / 経費 / 内訳 / 件数 をひと目で示す資料として利用可能
- 各画面の `.biz-note` 説明バナーが「この画面で何ができるか」を補助金申請担当者に明示

### 電子帳簿保存法
- レシート画像を `receiptDataUrl` として保存（本番では Google Drive 等への永続保管）
- 本番化時は タイムスタンプ・改竄検知 と組み合わせて電子帳簿保存法に対応可能
